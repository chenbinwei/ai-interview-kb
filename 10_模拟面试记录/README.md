## 2026-07-06：HSBC 项目追问 - Fallback Retrieval

日期：2026-07-06

模块：项目经历 / RAG 检索 / Agentic Workflow

问题：

> 你们的 Fallback Retrieval 是什么时候触发的？触发之后具体做了什么？

拆分追问：

```text
1. 什么情况说明第一次检索不够好？
2. fallback 会怎么改 query 或换检索方式？
3. fallback 之后怎么判断结果能不能用？
```

你的原始回答：

> 第一次检索不够好主要有三种情况：Top-K 结果为空；检索结果和输入术语明显不相关；关键术语没有命中，比如 MPF 没召回强积金。没有召回关键知识，或者召回内容置信度很低，就认为第一次检索失败。
>
> fallback 会做短语改写、同义词扩展、实体标准化和放宽检索条件，比如扩大 Top-K。如果有 Hybrid Search，会从纯向量检索切到关键词加向量联合检索。
>
> fallback 之后看三件事：是否召回目标术语，检索内容和输入语义是否一致，最终 JSON 校验是否通过。能覆盖关键术语、语义相关、并且生成结果通过结构校验，就认为 fallback 成功。

主要问题：

1. 结构是对的，已经能按“触发条件 -> fallback 动作 -> 成功判断”回答。
2. “定期存款 -> 定存”这类例子要小心，面试里更稳的是把口语/简称改成标准术语，比如“定存 -> 定期存款”。
3. Hybrid Search 如果不是项目里明确实现过，不要说死；可以说成后续优化方向或可选策略。

更强回答：

> Fallback Retrieval 主要在第一次检索结果不可靠时触发。比如 Top-K 为空、检索分数很低、结果和输入术语明显不相关，或者关键金融术语没有命中，比如用户说 MPF，但没有召回强积金或 Mandatory Provident Fund。
>
> 触发后，我会先做 query rewrite 和术语标准化，比如把“开户口”改写成“开立银行账户”，把“汇丰银行”标准化成 HSBC，把 MPF 扩展成 Mandatory Provident Fund / 强积金。然后再做二次检索，必要时扩大 Top-K。后续如果要进一步优化，也可以从纯向量检索扩展到关键词检索加向量检索的混合检索。
>
> Fallback 后不会直接相信结果，而是检查它有没有命中目标术语、语义是否和原句一致、是否有对应固定 Gloss，最后看生成的 HKSL Gloss JSON 能不能通过结构校验。如果这些都满足，才认为 fallback 成功；否则就标成低置信，进入人工或专家反馈。

## 2026-07-06：HSBC 项目追问 - JSON 校验失败怎么办

日期：2026-07-06

模块：项目经历 / 结构化输出 / 工程化

问题：

> 如果模型最后输出的 HKSL Gloss JSON 格式不合法，或者字段缺失，你们系统怎么处理？

拆分追问：

```text
1. 怎么发现 JSON 不合法？
2. 发现后是重新生成、自动修复，还是直接失败？
3. 如果连续几次都失败，怎么办？
```

你的原始回答：

> 这个我是真不懂，我只会肉眼看 JSON 格式，但其实也不太懂。

主要问题：

1. 需要区分“JSON 语法合法”和“业务 Schema 合法”。
2. 不能只靠 Prompt 要求模型输出 JSON，要有 parser、Schema 或 Code Node 做机器校验。
3. 重试必须有限制，连续失败要标记低置信或进入人工审核，避免死循环。

更强回答：

> 我会把 JSON 校验分成两层：第一层检查输出是不是合法 JSON，能不能被 parser 解析；第二层检查它是否符合业务 Schema，比如 gloss 是否是数组、unmatched_phrases 是否存在、confidence 是否是数字，必要字段是否完整。
>
> 如果校验失败，可以把具体错误反馈给模型，让它只修正 JSON，也可以用 Code Node 做一些确定性修复，比如补默认空数组。但如果涉及术语不确定或 Gloss 不可信，不能自动硬补。
>
> 如果连续 2-3 次重试仍然失败，就停止重试，把 case 标记为 low confidence 或 needs review，记录原始输入、模型输出和校验失败原因，进入人工或专家反馈，不能让错误 JSON 进入下游动画系统。

## 2026-07-06：HSBC 项目追问 - Fallback 失败后怎么办

日期：2026-07-06

模块：项目经历 / 幻觉控制 / Human-in-the-Loop

问题：

> 如果 fallback 也没有检索到合适结果，你的系统应该怎么办？模型能不能自己编一个 Gloss？

你的原始回答：

> 我感觉只能靠人工来补充库了。

主要问题：

1. 方向是对的，说明你知道不能让模型在金融术语和固定 Gloss 上自由发挥。
2. 还需要补工程化流程：系统不能静默失败，要标记低置信、记录未命中短语、进入专家审核。
3. “能不能自己编”要说清楚边界：可以生成候选草稿辅助专家，但不能当作正式 Gloss 自动进入下游动画。

更强回答：

> 如果 fallback 之后仍然没有检索到合适结果，我不会让模型直接编一个正式 Gloss。因为这个项目面向金融无障碍场景，金融术语和固定 Gloss 必须以术语库、规则库和专家反馈为准。
>
> 系统上我会把这类 case 标记为 low confidence 或 needs review，同时记录 unmatched phrase、原始输入、改写后的 query、Top-K 检索结果和模型输出。下游 JSON 可以保留结构，但对未确认术语加标记，避免它直接进入动画生成。
>
> 模型可以做一件事：生成一个 candidate Gloss 作为专家参考，但这个候选不能直接作为最终结果。专家确认后，再把标准 Gloss、别名和示例补回术语库或规则库，并把这个 case 加入测试集，避免下次再失败。
