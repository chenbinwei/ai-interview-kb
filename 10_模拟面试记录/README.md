## 2026-07-06：HSBC 项目追问 - Fallback Retrieval

日期：2026-07-06

模块：项目经历 / RAG 检索 / Agentic Workflow

问题：

> 你们的 Fallback Retrieval 是什么时候触发的？触发之后具体做了什么？

拆分追问：

```text
1. 什么情况说明第一次检索不够好？
2. fallback 会怎么改 query 或换检索方式？
3. fallback 之后怎么判断结果能不能用？
```

你的原始回答：

> 第一次检索不够好主要有三种情况：Top-K 结果为空；检索结果和输入术语明显不相关；关键术语没有命中，比如 MPF 没召回强积金。没有召回关键知识，或者召回内容置信度很低，就认为第一次检索失败。
>
> fallback 会做短语改写、同义词扩展、实体标准化和放宽检索条件，比如扩大 Top-K。如果有 Hybrid Search，会从纯向量检索切到关键词加向量联合检索。
>
> fallback 之后看三件事：是否召回目标术语，检索内容和输入语义是否一致，最终 JSON 校验是否通过。能覆盖关键术语、语义相关、并且生成结果通过结构校验，就认为 fallback 成功。

主要问题：

1. 结构是对的，已经能按“触发条件 -> fallback 动作 -> 成功判断”回答。
2. “定期存款 -> 定存”这类例子要小心，面试里更稳的是把口语/简称改成标准术语，比如“定存 -> 定期存款”。
3. Hybrid Search 如果不是项目里明确实现过，不要说死；可以说成后续优化方向或可选策略。

更强回答：

> Fallback Retrieval 主要在第一次检索结果不可靠时触发。比如 Top-K 为空、检索分数很低、结果和输入术语明显不相关，或者关键金融术语没有命中，比如用户说 MPF，但没有召回强积金或 Mandatory Provident Fund。
>
> 触发后，我会先做 query rewrite 和术语标准化，比如把“开户口”改写成“开立银行账户”，把“汇丰银行”标准化成 HSBC，把 MPF 扩展成 Mandatory Provident Fund / 强积金。然后再做二次检索，必要时扩大 Top-K。后续如果要进一步优化，也可以从纯向量检索扩展到关键词检索加向量检索的混合检索。
>
> Fallback 后不会直接相信结果，而是检查它有没有命中目标术语、语义是否和原句一致、是否有对应固定 Gloss，最后看生成的 HKSL Gloss JSON 能不能通过结构校验。如果这些都满足，才认为 fallback 成功；否则就标成低置信，进入人工或专家反馈。
