# 自主智能行业与创始人面试速记

这个文件用于准备“自主智能 / Agentic AI / 具身智能 / AI 机器人”方向的创始人技术面。目标不是把自己包装成强化学习或机器人专家，而是能讲清楚行业问题、技术闭环、基础概念，以及自己的经历如何切入。

## 一句话定位

> 这家公司的“自主智能”大概率指的是以大模型 / Agent 为核心，让机器人或 AI 系统具备自主理解任务、规划行动、调用工具或控制设备、根据环境反馈自我修正的能力，方向更接近 Agentic AI + 具身智能 / 机器人，而不只是普通聊天机器人。

更口语：

> 他们想做的不是“会聊天的 AI”，而是“能自己判断下一步该怎么做，并在环境里执行任务的 AI / 机器人系统”。

## 自主智能是什么

### 15 秒版

> 自主智能不是单纯生成答案，而是让 AI 系统围绕目标自主完成感知、理解、规划、行动、反馈和修正的闭环。Agent 是实现自主智能的一种方式，但真正的自主智能还要求更强的环境交互、长期记忆和持续学习能力。

### 60 秒版

> 我理解自主智能不是简单的聊天机器人，而是一个能围绕目标形成闭环的 AI 系统。普通 Chatbot 更多是用户问一句、模型答一句；Agent 会更进一步，可以根据目标拆解任务、选择工具、观察结果，再决定下一步。
>
> 但自主智能比普通 Agent 还要更大一点。它不只是会调用工具，而是要具备感知环境、理解状态、规划行动、根据反馈修正，甚至长期积累经验的能力。
>
> 所以如果一句话概括，我会说：Chatbot 偏对话，Agent 偏任务执行，自主智能偏目标驱动的持续决策和反馈学习。真正难的不是让模型生成一句话，而是让系统在复杂环境中稳定、可控、可评估地完成任务。

## Chatbot / Agent / 自主智能

| 概念 | 核心 | 特点 |
|---|---|---|
| Chatbot | 对话回复 | 用户问一句，模型答一句 |
| Agent | 任务执行 | 能规划、调工具、观察结果、调整步骤 |
| 自主智能 | 目标驱动闭环 | 感知环境、理解状态、行动、反馈、学习和泛化 |

一句话：

> Agent 是实现形式，自主智能是更大的目标。

## 行业现在的问题

> 自主智能现在处在从 demo 到可靠系统的阶段。大模型让系统具备了语言理解、推理和工具调用能力，但真正难的是闭环：感知是否可靠、规划是否稳定、执行是否可控、反馈是否能转化为学习、评估是否能证明系统真的完成任务。

真正要解决的问题：

```text
1. 任务规划：目标怎么拆成可执行步骤？
2. 环境理解：系统如何知道当前状态？
3. 行动执行：工具 / API / 机器人动作能不能可靠执行？
4. 反馈学习：失败经验怎么沉淀，下次能不能少犯错？
5. 评估验证：怎么证明不是偶然成功？
6. 安全边界：哪些动作必须人工确认？
```

## 核心模块怎么说

如果问“一个自主智能系统有哪些模块”，可以答：

```text
1. 模型层：LLM / 多模态模型，负责理解、推理和生成。
2. 感知层：文本、视觉、传感器或环境状态输入。
3. 状态层：维护当前任务进度、上下文、工具结果和环境状态。
4. 规划层：任务拆解、步骤排序、动态重规划。
5. 执行层：调用工具、API，或控制机器人动作。
6. 记忆层：短期上下文、长期经验、技能库。
7. 评估层：判断结果是否达标，是否需要重试或转人工。
8. 安全层：权限控制、风险动作审核、日志和可追踪性。
```

面试话术：

> 我不会只把它看成一个大模型，而会看成一个闭环系统。模型提供推理能力，但系统能不能可靠，取决于状态管理、工具执行、规划反馈、评估和安全边界。

## 和 HSBC 项目怎么接

### 稳妥说法

> 我的 HSBC 手语项目不是完整意义上的自主智能，因为它没有真实环境中的连续行动和长期学习。但它有一些相似的系统思想：不是一次性 LLM 生成，而是通过预处理、规则约束、术语检索、fallback、结构校验和专家反馈来修正输出。
>
> 这个过程体现了一个闭环思想：模型输出要被外部知识、规则和反馈机制约束。如果往自主智能方向扩展，我理解就需要加入更明确的状态管理、任务规划、自动评估、长期记忆和人机协同机制。

### 不要说太满

不要说：

```text
我做过完整自主智能系统。
我做过强化学习训练。
我做过机器人控制。
```

更稳：

```text
我做过 LLM / RAG / Agentic Workflow 的实际项目，理解模型输出如何被知识、规则、校验和人类反馈约束；完整自主智能还需要补规划、RL、长期记忆和环境交互。
```

## 数学基础为什么重要

面试话术：

> 自主智能涉及的不只是生成文本，而是状态空间里的决策和优化。所以数学基础很重要。线性代数对应 embedding、attention、低秩分解；概率统计对应 softmax、采样、不确定性和 KL 散度；优化对应 loss、梯度下降和模型训练；强化学习对应 state、action、reward、policy 这些决策建模。

最低要会的词：

| 方向 | 要会说什么 |
|---|---|
| 线性代数 | 向量、矩阵、相似度、低秩分解 |
| 概率统计 | softmax、采样、不确定性、KL 散度 |
| 优化 | loss、梯度下降、学习率、过拟合 |
| 强化学习 | state、action、reward、policy、value |

## 强化学习基础

### 15 秒版

> 强化学习是让智能体在环境中通过试错学习策略。它在某个状态下选择动作，环境返回奖励和下一个状态，目标是最大化长期累计奖励。

### 关键概念

| 词 | 中文 | 怎么理解 |
|---|---|---|
| State | 状态 | 当前环境和任务进度 |
| Action | 动作 | 智能体可以执行的选择 |
| Reward | 奖励 | 环境对动作结果的反馈 |
| Policy | 策略 | 在某个状态下该选什么动作 |
| Value | 价值 | 某个状态或动作长期来看有多好 |
| Exploration | 探索 | 尝试新动作 |
| Exploitation | 利用 | 使用当前已知最优动作 |

和自主智能的关系：

> 自主智能要做决策，不只是生成答案，所以强化学习里的状态、动作、奖励和策略是很核心的建模方式。

## RLHF / DPO 和自主智能

> RLHF 和 DPO 都是用人类偏好来调整模型行为。它们不是让模型直接获得机器人行动能力，但能让模型更符合人类偏好，比如更安全、更有帮助、更少胡说。对自主智能系统来说，人类反馈可以作为系统改进的一部分。

结合自己项目的边界：

> 我们项目里的专家反馈主要用于规则库、知识库、prompt 和测试集迭代，并没有直接更新模型权重，所以它更像 human feedback driven 的系统迭代，不是严格意义上的 RLHF 或 DPO。

## 如果提 Nature 怎么接

不要质疑，也不要盲目附和。用技术问题承接：

> 我想请教一下，公司希望冲击高水平论文的核心科学问题是什么？是自主智能的学习机制、长期记忆、规划决策，还是具身环境里的泛化能力？

继续问：

> 目前有没有明确的 benchmark 或实验环境？比如任务完成率、泛化能力、样本效率、长期记忆能力这些指标会怎么评估？

判断对方是否有实质：

```text
更实：具体任务、benchmark、实验平台、baseline、核心算法、评价指标。
更虚：颠覆行业、重构未来、全球领先、范式革命，但缺少实验细节。
```

## 创始人高频追问

### 1. 你怎么理解自主智能？

> 自主智能不是单纯聊天，而是目标驱动的闭环系统。Chatbot 是问答，Agent 能拆任务和调工具，而自主智能更强调感知环境、规划行动、观察反馈和持续修正。我的项目还不是完整自主智能，但它有类似的闭环思想：通过检索、规则、校验和专家反馈约束模型输出。所以我理解这个方向的关键，不只是模型强，而是系统能不能可靠地规划、执行、评估和改进。

### 2. 一个自主智能系统最核心的模块有哪些？

> 我会拆成模型、感知、状态、规划、执行、记忆、评估和安全八块。模型负责理解和推理，感知层把环境信息输入进来，状态层记录当前任务进度，规划层决定下一步，执行层调用工具或控制设备，记忆层沉淀经验，评估层判断是否达标，安全层控制权限和人工审核。

### 3. 行业现在最大的瓶颈是什么？

> 我觉得不是模型会不会说话，而是系统能不能稳定做事。现在很多 Agent demo 看起来很好，但生产里会遇到规划跑偏、工具调用失败、反馈无法沉淀、评估不可靠、安全边界不清楚等问题。所以行业从 demo 到落地，核心是可靠性、可控性和可评估性。

### 4. 你的经历如何切入？

> 我更适合从 LLM 应用系统、Agentic Workflow、评估反馈和数据闭环切入。HSBC 项目让我接触到模型输出如何被知识库、规则、结构校验和专家反馈约束；电池项目让我有序列建模和模型评估经验。完整自主智能里的 RL、控制和具身交互我还需要继续补，但我能先在系统构建、评估和反馈迭代上发挥作用。

### 5. 如果让你设计一个可控 Agent，你会怎么做？

> 我会先避免完全开放自主，而是做可控的 Agentic Workflow：固定关键流程，局部让模型判断；每一步保留状态和日志；工具调用用 schema 约束参数；失败时允许有限次数重试和重规划；高风险动作必须人工确认；最后用固定测试集、任务成功率、人工评估和 bad case 分析来持续迭代。
