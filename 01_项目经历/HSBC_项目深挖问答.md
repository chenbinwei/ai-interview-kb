# HSBC 项目深挖问答

这个文件专门应对面试官深挖。原则是：能确定的就直接说；不确定的用“项目阶段/建议口径/可以理解为”来表达，不把未核实的细节说死。

## 1. Dify 里具体有几个节点？每个节点做什么？

建议回答：

> 整个 Workflow 可以理解为 7-8 个核心节点：Start、Text Preprocess、HKSL Grammar Convert、Knowledge Retrieval、Fallback Retrieval、Function / Code Node、Aggregation 和 JSON Validation。实际项目里我更关注链路逻辑，而不是节点数量本身。

节点链路：

```text
Start
  ↓
Text Preprocess
  ↓
HKSL Grammar Convert
  ↓
Knowledge Retrieval
  ↓
Fallback Retrieval
  ↓
Function / Code Node
  ↓
Aggregation
  ↓
JSON Validation
  ↓
Output
```

各节点作用：

| 节点 | 面试说法 |
|---|---|
| Start | 接收用户原始中文/粤语文本 |
| Text Preprocess | 去除无效字符、标准化标点、简单实体归一化 |
| HKSL Grammar Convert | 把中文语序转换成更接近香港手语的表达顺序 |
| Knowledge Retrieval | 检索金融术语、固定 Gloss 和专业表达 |
| Fallback Retrieval | 对未命中的短语做 phrase rewrite、synonym search 和二次检索 |
| Function / Code Node | 处理变量替换、Gloss 拼接和 JSON 生成 |
| Aggregation | 整合 LLM 输出、检索结果和函数结果 |
| JSON Validation | 校验 JSON Schema、字段完整性和格式错误，失败时重新生成 |

## 2. 规则库和术语库是什么格式？

建议回答：

> 初期更像业务知识表，主要用 Excel 或 CSV 维护，方便 HSBC、DPO 和手语专家直接看、改、加备注。后续会导入 Dify Knowledge Base，用于规则检索和术语检索。

规则库例子：

```json
{
  "rule_id": "R001",
  "description": "时间信息优先",
  "example": "今天 我 银行 办理业务"
}
```

术语库例子：

```json
{
  "source": "银行账户",
  "target": "BANK_ACCOUNT",
  "domain": "finance",
  "notes": "优先使用固定 Gloss"
}
```

不要主动说：

```text
我们用了 Milvus。
```

更稳的说法：

> 项目阶段主要用 Dify 内置知识库，后续如果数据规模变大或需要更细的检索控制，可以考虑 Milvus 或 pgvector。

## 3. 检索用 Dify 内置知识库，还是外部向量库？

建议回答：

> 主要使用 Dify 内置知识库。原因是这个阶段更偏业务可行性验证，数据规模不大，Dify 内置知识库已经够用，而且方便非工程背景的业务人员维护规则和术语。

如果追问为什么不用外部向量库：

> 外部向量库适合数据规模更大、检索策略更复杂、需要更强索引控制的场景。这个项目初期主要问题不是向量库性能，而是规则、术语、Prompt 和专家反馈闭环，所以优先用维护成本更低的方案。

## 4. 有没有固定测试集？大概多少条？

稳妥回答：

> 有一批固定 case set 用于回归测试，覆盖金融术语、政务术语、长句、多实体句和容易歧义的表达。数量可以按 100-200 条这个量级理解，但它更像项目内部测试集，不是公开 benchmark。

用途：

- Prompt 回归测试
- JSON 稳定性测试
- 检索覆盖率测试
- 专家可用性评估

如果不确定数量：

> 我不想把数量说得太死，我记得是百条量级，主要用于比较不同 Prompt、检索策略和模型组合的稳定性。

## 5. 专家反馈怎么记录？

建议回答：

> 初期反馈可能来自会议记录、批注和专家口头意见，后期更适合沉淀成 Excel 问题表。表里会记录原句、模型输出、专家修订、错误类型和备注。

字段：

```text
原句
模型输出
专家修订
错误类型
备注
```

错误类型：

- Gloss 错误
- 语序错误
- JSON 格式错误
- 术语错误
- 检索未命中

面试补一句：

> 这样做的好处是每条 bad case 都能反推到具体问题：是 Prompt 不够约束、检索没召回、术语库缺词，还是输出格式不稳定。

如果追问“专家具体怎么打分”：

> 早期我们会让内部有手语背景的同事先做一轮评估，偏三分制，比如 2 表示好、1 表示勉强可用、0 表示不好，同时尽量标注错误类型，比如术语错误、语序错误、Gloss 错误或格式问题。后面为了让指标更稳定，也会收敛成专家二分类，也就是可用 / 不可用，重点判断输出能不能进入后续动画流程，或者是否需要人工重做。

如果追问“有时候同事只能打分，不能标错误类型怎么办”：

> 这是正常的。不是每个评估人员都能准确判断错误来源，所以我会把评分和错误类型分开看。评分用于统计可用率，错误类型用于定位问题。如果只拿到了 0/1/2 或可用/不可用，也可以先记录下来；后续再结合模型输出、检索结果和专家意见，补充错误归因。

如果追问“专家反馈怎么回流到系统”：

> 专家和手语同事的反馈不会直接让模型自动改规则。更稳的流程是：先把 bad case、专家修订和评分整理出来，让 LLM 辅助归纳常见错误模式和候选规则，比如哪些语序经常错、哪些术语需要固定 Gloss；然后这些候选规则再交给手语同事或专家确认。确认后的规则才会补进规则库、术语库、Prompt 或测试集。

## 6. 可用比例提升 20%+ 怎么统计？

先定义“可用”：

> 可用是指输出满足 JSON 正确、核心术语正确、没有明显语序错误，并且不需要人工重做，可以进入后续动画生成或只需要少量修改。

统计方式：

```text
Baseline：Prompt 直接生成
After：加入规则检索、术语检索、Fallback Retrieval 和 JSON 校验
```

示例口径：

> 比如直接生成时专家认为可用的比例大概在 55% 左右，加入检索、规则和 fallback 后提升到 78% 左右，所以可以说可用比例提升约 20%+。

更保守口径：

> 具体数字我不会说得过死，但提升来自专家可用性评估。我们比较的是优化前后同一批 case 中，输出能否直接进入后续流程或只需要少量修改。

补充口径：

> 这里的专家可用性评估可以分阶段理解。早期可以是内部手语同事按 0/1/2 三分制打分，2 是好，1 是勉强可用，0 是不可用；后续为了统计更清晰，可以收敛成专家二分类，也就是可用 / 不可用。统计 20%+ 时，核心是比较同一批测试 case 在 baseline 和优化后，被判断为可用的比例变化。

如果自动指标和专家反馈不一致：

> 我会以专家反馈为准。BLEU、ROUGE、JSON 合法率、术语命中率这些自动指标可以帮助批量筛查和定位问题，但手语项目最终要看专家认为输出是否符合 HKSL 表达、是否能服务后续动画生成。所以自动指标是辅助，专家可用性判断是主标准。

## 7. 你具体做了哪些工作？分别占多少？

建议拆分：

```text
Prompt 设计：35%
知识库整理：20%
Workflow 搭建：20%
测试与评估：15%
Code Node / Function：10%
```

面试回答：

> 我的主要工作是 Prompt 约束设计、知识库整理和 Workflow 配置，也参与了 Fallback Retrieval、JSON 校验和测试 case 的整理。代码部分更多是辅助性质，比如变量处理、格式校验和结果拼接，核心价值还是把业务规则、检索和生成流程串成一个可迭代的工作流。

## 8. Embedding 模型怎么评估？

短答：

> Embedding 不能只说“哪个模型口碑好”，也不能完全靠肉眼看。更稳的方式是先用检索指标筛选，再用下游 Gloss 质量和专家反馈判断。

展开回答：

> 在这个项目里，Embedding 主要影响的是术语库、规则库和近义短语的召回。比如用户说“开户口”，系统能不能在 Top-K 结果里召回“开立银行账户”或对应的 `ACCOUNT_OPEN`。所以我会准备一批 query-target 测试样例，比如口语表达、粤语表达、英文缩写、金融术语和标准 Gloss 的对应关系，然后比较不同 Embedding 模型的 Top-K 命中情况。
>
> 可以看的指标包括 Hit@K / Recall@K，也就是正确术语有没有出现在前 K 条结果里；也可以看排名，比如正确结果是不是排在 Top-1、Top-3，而不是很靠后。除了自动指标，还要看误召回：比如两个金融术语语义接近但不能替换，向量检索如果把它们混在一起，反而会影响最终 Gloss。
>
> 最后还是要看下游效果：换了 Embedding 后，术语命中率有没有提升，fallback 是否减少，专家判断可用率有没有提高。因为这个是手语和金融场景，最终不能只看向量相似度分数，要结合专家反馈。

面试口径：

> BGE-M3 这类模型本身在多语言检索上口碑比较好，但面试里我不会只说“它评价好所以用它”。我会说我们更关心它在项目测试集里的表现：中文、粤语、英文缩写和金融术语能不能召回正确的规则或 Gloss。指标上可以看 Top-K 命中率、Recall@K、正确结果排名和误召回情况；最终再结合专家对生成 Gloss 的可用性评价。

例子：

```text
query：开户口
目标召回：开立银行账户 / ACCOUNT_OPEN

query：MPF
目标召回：强积金 / Mandatory Provident Fund / MPF 对应 Gloss
```

容易踩坑：

> 不要说“Embedding 没法评估，只能人看”。更准确的是：检索阶段可以用指标评估，手语表达最终质量由专家评估。

## 常见追问

### 为什么不用 LangGraph？

> 项目初期目标是快速验证业务可行性，Dify Workflow 已经能满足多节点编排、检索和输出校验的需求。如果后续需要更复杂的状态管理、多 Agent 协作或更强的可观测性，我会考虑用 LangGraph 重构核心链路。

### 为什么不用外部向量库？

> 数据规模不大，Dify 内置知识库够用，也方便业务人员维护。外部向量库更适合大规模、多租户、复杂过滤和自定义检索策略的场景。

### 为什么不用微调？

> 这个项目的问题主要来自专业知识缺失、领域术语和结构化输出稳定性，而不是模型完全不会做。所以优先用 RAG、Prompt 和校验机制解决，比微调更快、更便宜，也更方便根据专家反馈迭代。

### 为什么叫 Agentic RAG？

> 我理解普通 RAG 一般是相对固定的流程：用户输入、检索 Top-K、拼进 Prompt、模型生成。这个项目虽然也用 RAG，但不是一次检索后直接生成，而是把任务拆成多个阶段：预处理、语法转换、规则检索、术语检索、fallback、函数/代码节点、聚合和 JSON 校验。系统会根据中间结果决定下一步，比如检索不到就做短语改写和二次检索，JSON 不合法就重新生成或修复，术语冲突时优先用术语库结果。所以我更愿意把它描述成 Agentic Workflow + RAG，也就是 Agentic RAG。

如果面试官说“这不就是复杂 RAG 吗”：

> 可以这么理解，RAG 本身确实可以做得很复杂，比如 query rewrite、rerank、hybrid search 都是 RAG 优化。但我这里强调 Agentic，是因为流程里有条件判断、fallback、校验和工具/代码节点，不是固定的一次检索生成。更准确地说，它是一个带 RAG 能力的多步骤工作流。

### 如果检索结果错了怎么办？

> 我会先看是 query 改写问题、知识库缺词、Embedding 召回问题，还是 Prompt 使用检索结果的方式不对。对应方案包括补术语、改 chunk 和 metadata、增加 fallback retrieval、提高检索 top-k 后再筛选，以及在输出校验里标记低置信结果。

### 怎么判断哪些步骤交给 LLM，哪些交给 RAG、规则或代码节点？

短答：

> 我会按任务性质分工：开放的语义理解交给 LLM，可靠知识交给 RAG，确定性格式和字段检查交给规则或代码。

展开回答：

> 像文本改写、语义理解、把普通句子转换成更接近 HKSL 语序的初稿，这类需要语言理解的步骤，可以交给 LLM，并用 Prompt 约束。
>
> 但像金融术语、固定 Gloss、香港手语语法规则，这些不能让模型自由发挥，所以要放到知识库里，通过 RAG 检索出来，再作为上下文约束模型。
>
> 另外，像 JSON 字段是否完整、格式是否合法、变量怎么拼接，这种确定性强、错了下游系统就不能用的部分，更适合用 Code Node 或规则校验，而不是让 LLM 自己判断。

面试总结：

```text
LLM：语义理解、句子改写、Gloss 初稿生成
RAG：手语规则、金融术语、固定表达
Code / Rule：JSON 校验、字段检查、变量拼接
专家反馈：判断输出是否真的可用
```

容易踩坑：

> 不要说“知识库交给 LLM 生成”。更稳的说法是：知识库来源是专家规则和术语表，LLM 可以辅助整理和格式化，但不能作为事实来源。

### 如果 LLM 生成的 Gloss 和检索到的术语不一致，怎么处理？

短答：

> 我会把术语库作为 source of truth。LLM 负责生成，但金融术语和固定 Gloss 以检索结果为准。

展开回答：

> 如果 LLM 生成的 Gloss 和检索到的术语不一致，我会把检索到的术语库结果作为优先级更高的依据。因为金融术语和固定 Gloss 不能让模型自由发挥。
>
> 处理上可以分三步：第一，在 Prompt 里明确要求模型优先使用检索到的术语；第二，在生成后做校验，检查输出里有没有使用术语库中的标准 Gloss；第三，如果发现不一致，就触发重新生成或用规则替换成标准术语。
>
> 如果还是不确定，就把这个 case 记录下来，交给手语专家确认，再更新术语库或 Prompt。

例子：

```text
术语库：信用卡 -> CREDIT_CARD
LLM 输出：BANK_CARD
处理：按术语库改回 CREDIT_CARD，或重新生成
```

面试总结：

```text
开放表达 -> LLM
固定术语 -> 知识库
格式正确 -> 校验节点
不确定 -> 专家反馈
```

### Fallback retrieval 是怎么知道这个词和那个词可以换着查的？

短答：

> 它不是凭空知道的，主要来自三类信息：术语库里的别名和同义表达、LLM 的 query rewrite，以及 Embedding 的语义相似召回。

展开回答：

> 第一种是最稳的方式：在术语库里维护标准词、别名和固定 Gloss。比如“银行账户”下面可以维护“户口、账户、银行户口、account”等别名。用户说“开户口”时，系统可以先映射到标准词，再去查固定 Gloss。
>
> 第二种是 query rewrite。如果第一次检索没有命中，可以让 LLM 把用户原始表达改写成更标准的检索词，比如把“开户口”改写成“开立银行账户”“申请银行账户”或“account opening”，再查一次。
>
> 第三种是 Embedding 语义召回。即使字面不一样，“开户口”“开立账户”“account opening”在向量空间里可能比较接近，所以可以召回相近候选。

边界：

> 在金融和手语场景里，不能完全依赖 Embedding，因为相似不代表可以替换。比如两个金融术语看起来接近，但对应的手语 Gloss 可能不同。所以更稳的策略是：术语库别名优先，LLM rewrite 和 Embedding 作为补充，最后再通过规则或 Prompt 判断召回结果能不能用。

例子：

```text
用户表达：开户口
术语库标准词：开立账户
别名：开户口 / 开户 / 银行账户 / account opening
固定 Gloss：ACCOUNT_OPEN
```

面试里可以这样收尾：

> 所以 fallback retrieval 的核心不是让模型随便猜同义词，而是用业务术语库做主约束，再用 query rewrite 和 embedding 提高召回率。

### Fallback Retrieval 什么时候触发？触发后怎么判断成功？

短答：

> Fallback Retrieval 会在第一次检索结果不可靠时触发，比如 Top-K 为空、检索分数低、结果明显不相关，或者关键金融术语没有命中。触发后会做 query rewrite、术语标准化、别名扩展和二次检索，最后用术语命中、语义相关性和 JSON 校验判断是否成功。

展开回答：

> 第一次检索不够好，通常有几个信号：第一，Top-K 结果为空；第二，检索结果和输入术语明显不相关；第三，关键术语没有命中。比如用户说 MPF，但没有召回强积金或 Mandatory Provident Fund，这就说明核心金融知识没有被召回。
>
> 触发 fallback 后，我会先做 query rewrite 和术语标准化。比如把“开户口”改成“开立银行账户”，把“汇丰银行”标准化成 HSBC，把 MPF 扩展成 Mandatory Provident Fund 或强积金。然后再做二次检索，必要时扩大 Top-K。后续如果要增强检索，也可以考虑从纯向量检索扩展到关键词检索加向量检索的混合检索。
>
> Fallback 后不会直接相信结果，还要看它是否命中目标术语、语义是否和原句一致、是否有对应固定 Gloss，最后生成的 HKSL Gloss JSON 是否能通过结构校验。如果这些都满足，才认为 fallback 成功；否则就标成低置信，进入人工或专家反馈。

面试总结：

```text
触发条件：Top-K 空 / 分数低 / 不相关 / 关键术语未命中
处理方式：query rewrite / 别名扩展 / 实体标准化 / 二次检索 / 扩大 Top-K
成功判断：术语命中 / 语义一致 / 固定 Gloss 可用 / JSON 校验通过
```

容易踩坑：

> 不要说 fallback 是让模型自由编一个 Gloss。金融术语和固定 Gloss 仍然以术语库、规则库和专家反馈为准。

### 如果 fallback 也没有检索到合适结果，模型能不能自己编 Gloss？

短答：

> 不能让模型直接编一个正式 Gloss。模型可以生成候选建议辅助专家，但金融术语和固定 Gloss 必须以术语库、规则库和专家反馈为准。

展开回答：

> 如果 fallback retrieval 之后仍然没有检索到合适结果，我不会让模型直接编一个正式 Gloss。因为这个项目面向 HSBC 金融无障碍场景，错误术语会影响下游手语动画，也会影响专业表达的可信度。
>
> 系统上我会把这类 case 标记为 low confidence 或 needs review，记录原始输入、unmatched phrase、改写后的 query、Top-K 检索结果和模型候选输出。JSON 可以保留结构，但对未确认术语做标记，避免它被当成可直接进入动画生成的结果。
>
> 模型可以生成 candidate Gloss 作为专家参考，但不能直接作为最终答案。专家确认后，再把标准 Gloss、别名、示例和错误类型补回术语库或规则库，并把这个 case 加入回归测试集。这样下一次遇到同类表达时，系统就能通过知识库或 fallback 正常命中。

面试总结：

```text
不能做：模型自由编正式 Gloss
可以做：生成候选草稿供专家参考
系统处理：低置信标记 / unmatched phrase 记录 / 暂停下游自动使用
后续闭环：专家确认 -> 补术语库/规则库 -> 加入测试集
```

容易踩坑：

> 不要说“查不到就让 LLM 自己生成”。更稳的说法是：开放语言表达可以让 LLM 参与，但固定术语和专业 Gloss 必须经过知识库或专家确认。

### 如果 HKSL Gloss JSON 格式不合法或字段缺失，系统怎么处理？

短答：

> 我会把 JSON 校验分成两层：第一层检查是不是合法 JSON，第二层检查是否符合业务 Schema，比如字段是否完整、类型是否正确。失败后可以把错误信息反馈给模型重新生成或用 Code Node 做简单修复；如果连续失败，就标记 low confidence，记录日志并进入人工审核，不能把错误 JSON 交给下游动画系统。

展开回答：

> 首先，JSON 校验不能只靠肉眼看。系统可以先用 parser 检查输出能不能被解析成 JSON；然后再用 JSON Schema、Pydantic 或 Dify 的校验/代码节点检查字段和类型。比如 gloss 应该是数组，unmatched_phrases 应该是数组，confidence 应该是数字，必要字段不能缺失。
>
> 如果只是简单格式问题，比如模型在 JSON 外面多输出了解释文字，或者少了某个字段，可以把校验错误反馈给模型，让它只修正 JSON，不重新发挥内容。也可以在 Code Node 里做一些确定性的修复，比如字段补默认空数组。
>
> 但如果涉及术语不确定、Gloss 不可信，不能靠自动修复硬补。连续 2-3 次重试仍失败，就应该停止重试，标记为 low confidence 或 needs review，记录原始输入、模型输出、错误类型和校验失败原因，交给人工或专家反馈。

面试总结：

```text
发现方式：JSON parser / JSON Schema / Pydantic / Code Node
检查内容：语法合法 / 必填字段 / 字段类型 / 业务规则
失败处理：错误反馈给模型重生成 / Code Node 简单修复 / 有限重试
兜底策略：连续失败 -> low confidence -> 人工审核 / 专家反馈
```

容易踩坑：

> 不要只说“Prompt 里要求输出 JSON”。Prompt 约束不等于工程校验。面试里要强调下游系统需要机器可解析、可验证的结构化输出。

如果追问“怎么避免一直重试”：

> 我会设置最大重试次数，比如 2-3 次，避免模型反复生成进入死循环。每次重试不是简单重复问，而是把具体校验错误反馈给模型，比如 `gloss` 应该是数组但当前是字符串，让它只修正 JSON 结构。如果连续失败，就停止自动重试，标记 low confidence 或 needs review，并记录 bad case。

如果追问“什么能自动修，什么不能自动修”：

> 格式类问题可以自动修，比如 JSON 外面多了说明文字、缺少 `unmatched_phrases` 默认空数组、字段类型比较明确地写错。内容类问题不能硬修，比如 Gloss 术语不确定、金融术语没命中、HKSL 语序不合理，或者检索结果和最终输出冲突。这些要回溯到检索、知识库、Prompt 或专家反馈环节处理。

如果追问“既然前面内容对了，为什么后面还会出问题”：

> 因为前面解决的是内容大方向，比如术语和 Gloss 是否合理；后面要保证这些内容被包装成下游动画系统可用的稳定 JSON。即使前面知道“开户口”对应 `ACCOUNT_OPEN`，最后也可能输出成字符串而不是数组，或者在 JSON 外面多输出解释文字，导致机器不能解析。所以后置校验不是重新做全部语义判断，而是确认正确内容没有在生成、聚合和格式化过程中丢失或变形。

一句话总结：

> 前面负责内容尽量对，后面负责输出一定能用。

### Parser、Schema、Code Node 分别在这里做什么？

短答：

> Parser 负责把模型输出读成 JSON，Schema 负责检查字段和类型，Code Node 负责做确定性的修复、拼接和重试判断。

展开回答：

> 在这个项目里，模型输出本质上先是一段文本。Parser 做的是第一层检查：这段文本能不能被机器解析成合法 JSON。比如有没有多余解释文字、引号有没有缺失、括号有没有闭合。
>
> Schema 是第二层检查：即使它是合法 JSON，也要看字段是不是符合业务要求。比如 `gloss` 应该是数组，`confidence` 应该是数字，`unmatched_phrases` 应该存在并且是数组。
>
> Code Node 则更偏工程处理。它可以补一些默认字段、拼接检索到的标准 Gloss、把校验错误返回给模型重试，或者在连续失败时标记 low confidence，避免错误结果直接进入下游动画生成。

如果追问“这个是人工检查吗”：

> 不是每条都靠人工肉眼检查。更准确地说，是人先定义规则和 Schema，比如哪些字段必填、字段类型是什么、confidence 范围是多少，然后系统用 JSON parser、JSON Schema、Pydantic 或 Code Node 自动校验。人工或专家主要处理两类问题：一类是校验连续失败的 bad case，另一类是 Gloss 语义、术语和手语表达是否真的正确。

如果追问“Parser 有哪些方法”：

> 最基础的是标准 JSON parser，比如 Python 的 `json.loads()` 或 JavaScript 的 `JSON.parse()`，它们负责判断输出是不是合法 JSON。如果模型多输出了解释文字，可以先提取 JSON 块再解析，但这只是补救。更稳的方式是结合 JSON Mode、Structured Outputs 或 Pydantic，让模型输出尽量符合固定结构，再由 Schema 自动验证。

面试总结：

```text
Parser：检查能不能读
Schema：检查结构对不对
Code Node：做稳定的工程处理
```

一句话收尾：

> 所以我理解它们不是三个孤立概念，而是结构化输出的三层保护：先能解析，再符合格式，最后用代码节点处理异常和兜底。
