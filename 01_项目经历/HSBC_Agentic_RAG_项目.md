# HSBC 香港手语无障碍 Agentic RAG 项目

## 一句话定位

这是一个面向无障碍内容生成的 Agentic RAG 项目：系统把普通文本转换成结构化 HKSL Gloss，并通过规则检索、术语检索、Prompt 约束和专家反馈，提高后续手语动画生成的可用性。

## 2 分钟面试版

我在 Glassbox AI 参与了香港上海汇丰银行 HSBC 的香港手语无障碍项目，目标是把普通文本自动转换成结构化 HKSL Gloss，支持后续手语动画生成和无障碍内容展示。

这个任务不是简单中英翻译，因为香港手语有自己的语序、语法规则和领域表达方式。我们基于 Dify 搭建了一个 Agentic RAG 工作流，把任务拆成句子修正、内容缩减、规则检索与翻译、术语和短语检索、结果校验几个步骤。

我参与整理香港手语语法规则和领域词汇库，对比 GPT-5、Gemini 2.5、Claude 等大模型，以及 BGE-M3、nomic-embed-text-1.5 等 Embedding 模型在翻译和近义词匹配中的效果。我们根据手语专家反馈不断优化规则、检索策略和 Prompt，使生成结果可用比例提升约 20%+。

这个项目让我比较完整地接触了 AI 应用从规则知识整理、RAG 检索、LLM 生成、评估反馈到工作流优化的过程。后续如果工程化上线，我会补齐接口服务、向量库管理、日志监控、评估集、缓存和人工兜底机制。

## 项目背景

HSBC 需要支持香港手语无障碍内容展示。普通文本不能直接生成高质量手语动画，需要先转换成符合香港手语语法和表达习惯的结构化 HKSL Gloss。

难点在于：
- 手语 Gloss 不是普通自然语言翻译。
- 需要符合手语语序、语法规则和领域词汇。
- 同一句普通文本可能需要先简化、改写，再翻译。
- 结果质量需要由手语专家判断，不能只靠通用文本指标。

## 业务目标

核心目标：
- 把普通文本转换成结构化 HKSL Gloss。
- 提高输出对手语动画生成的可用性。
- 降低人工改写和专家校验成本。

可表达指标：
- 生成结果可用比例提升约 20%+。
- 支持后续手语动画生成和无障碍内容展示。
- 通过专家反馈持续优化规则和检索策略。

## 整体架构

```text
用户/业务文本
  ↓
文本预处理与句子修正
  ↓
内容缩减与表达简化
  ↓
香港手语规则检索
  ↓
领域术语/短语检索
  ↓
LLM 生成结构化 HKSL Gloss
  ↓
规则校验与结果校验
  ↓
专家反馈
  ↓
规则库、术语库、Prompt、检索策略迭代
```

## Dify 工作流怎么讲

可以这样说：

> 我们用 Dify 主要是为了快速搭建可视化的 Agentic Workflow，把一个复杂翻译任务拆成多个可控节点。每个节点只负责一个子任务，比如句子修正、内容压缩、规则检索、术语检索、翻译生成和结果校验。这样比单次 LLM 调用更可控，也更方便根据专家反馈定位是哪一步出了问题。

可补充：
- Dify 适合快速验证流程。
- 真正工程化时可以把核心链路拆成后端服务。
- Dify 原型不等于只会拖节点，关键是对任务拆解、检索策略、Prompt 约束和评估闭环的理解。

## 为什么叫 Agentic RAG

普通 RAG 主要是：

```text
用户问题 → 检索知识 → 拼接上下文 → LLM 回答
```

这个项目更接近 Agentic RAG，因为它不是一次检索后直接回答，而是把任务拆成多个步骤：

```text
修正 → 缩减 → 查规则 → 查术语 → 翻译 → 校验 → 反馈优化
```

面试版说法：

> 它不是简单的知识库问答，而是一个带任务拆解的 RAG 工作流。系统会在不同阶段检索不同类型的知识，比如语法规则和术语短语，再让 LLM 按约束生成结构化 Gloss，最后进行校验。所以我会把它称为 Agentic RAG 或 Agentic Workflow + RAG。

## RAG 检索链路

本项目里的 RAG 不只是查文档，而是查两类知识：

1. 规则知识
   - 香港手语语法规则
   - 句子结构转换规则
   - 表达顺序约束

2. 术语/短语知识
   - 领域词汇
   - 高频表达
   - 近义词匹配
   - 特定业务语境下的 Gloss 表达

可说的流程：

```text
输入文本
  ↓
抽取关键词和语义意图
  ↓
Embedding 检索相近规则/术语
  ↓
把检索结果作为上下文放入 Prompt
  ↓
LLM 生成 HKSL Gloss
  ↓
校验是否符合规则和术语表
```

## 模型与 Embedding 对比怎么讲

简历里写了：
- LLM：GPT-5、Gemini 2.5、Claude
- Embedding：BGE-M3、nomic-embed-text-1.5

面试版说法：

> 我们主要从两个维度比较模型：一是 LLM 对手语规则和结构化输出的遵循能力，二是 Embedding 在术语和近义短语匹配上的召回效果。LLM 侧关注输出是否符合 Gloss 格式、是否保留核心语义、是否容易幻觉；Embedding 侧关注相似术语能不能被召回，以及召回结果是否对翻译有帮助。

如果被追问“怎么评估”：

> 我会把评估拆成自动评估和专家评估。自动评估看格式合法率、术语命中率、规则匹配率、检索命中率；专家评估看生成 Gloss 是否可用、是否需要大幅修改、是否能进入后续动画生成流程。

## 最大技术难点

建议你主打这个难点：

> 最大难点是普通文本到 HKSL Gloss 不是简单翻译，而是一个受规则约束的结构化生成任务。

展开：
- 普通文本可能太长，需要先压缩。
- 原句语序不一定符合手语表达。
- 通用 LLM 容易生成自然语言解释，而不是结构化 Gloss。
- 手语规则和领域术语需要被准确引用。
- 质量标准依赖专家反馈，不能只看 BLEU/ROUGE 这类文本相似度。

## 解决方案

可以按四层讲：

1. 任务拆解
   - 不让 LLM 一步完成所有事。
   - 拆成修正、缩减、规则检索、术语检索、翻译、校验。

2. RAG 约束
   - 检索手语规则和领域术语。
   - 用检索结果限制模型输出。

3. Prompt 约束
   - 明确输出格式。
   - 要求只输出 Gloss。
   - 要求优先使用术语库表达。

4. 专家反馈闭环
   - 收集专家修改意见。
   - 更新规则库、术语库、Prompt 和检索策略。

## 幻觉控制怎么讲

面试版：

> 我主要从输入、检索、生成和校验四个环节控制幻觉。输入侧先做句子修正和内容缩减，降低歧义；检索侧提供手语规则和术语库作为依据；生成侧用 Prompt 限制输出格式和可用词汇；校验侧检查结果是否符合规则、是否使用了正确术语，并结合专家反馈迭代。

可以补一句：

> 对这类项目来说，幻觉不只是编造事实，也包括生成了不符合手语规则的表达，所以我们更关注规则遵循和专家可用性。

## 高频追问

### 1. 为什么不用普通翻译模型？

因为 HKSL Gloss 不是普通自然语言翻译。它需要符合手语语法、语序和领域术语表达，普通翻译模型容易输出自然语言解释，不能直接用于后续动画生成。

### 2. 为什么不用普通 RAG？

普通 RAG 更像一次检索加回答，但这个任务需要多步处理，包括修正、缩减、规则检索、术语检索、生成和校验，所以更适合 Agentic Workflow + RAG。

### 3. Dify 在项目里起什么作用？

Dify 负责把多个 LLM 和检索节点编排成可视化工作流，方便快速验证流程、定位问题和迭代 Prompt。核心不是工具本身，而是任务拆解、检索策略和反馈闭环。

### 4. 可用比例提升 20%+ 怎么解释？

可以说：

> 这里的可用比例主要来自专家反馈，指生成结果经过较少修改或可以进入后续流程的比例。我们通过规则库、术语库、Prompt 和检索策略迭代，让专家认为可用的输出占比提升了约 20%+。

如果面试官继续追问统计口径，要诚实说明：

> 这个指标更偏业务可用性评估，不是单一自动指标。更严谨的做法是建立固定测试集，按格式合法率、术语命中率、规则匹配率和专家通过率分别统计。

### 5. 你个人贡献是什么？

建议回答：

> 我的主要贡献包括三部分：第一，参与整理香港手语语法规则和领域词汇库；第二，参与对比不同 LLM 和 Embedding 模型在翻译和近义词匹配中的效果；第三，根据手语专家反馈优化规则、检索策略和 Prompt，使生成结果可用比例提升约 20%+。
