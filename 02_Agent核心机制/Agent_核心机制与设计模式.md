# Agent 核心机制与设计模式

这个文件用于准备 Agent 高频面试题。重点不是把 Agent 讲得很玄，而是能说清楚：Agent 和 Chatbot 的区别、什么情况下算更自主、Agent 有哪些常见运行模式，以及生产系统为什么通常追求“可控的半自主”。

## Agent 是什么？

### 15 秒版

> Agent 是一个围绕目标执行任务的 LLM 系统。它不只是回复用户，而是能规划步骤、调用工具、观察结果，并根据中间结果继续调整行动。

### 60 秒版

> Chatbot 更偏对话系统，核心是根据用户输入生成回复。它可以支持多轮对话，也可以接文件、图片或简单工具，但整体还是以“回答用户”为主。
>
> Agent 更偏任务执行系统。它会围绕一个目标判断下一步怎么做，比如要不要查知识库、要不要调用工具、工具结果够不够、是否需要继续重试或转人工。典型流程是 Plan、Act、Observe，再根据观察结果调整。
>
> 所以两者的核心区别不是有没有 LLM，而是谁控制流程。Chatbot 更像输入到输出；Agent 是有一个行动闭环。

## Agent 和 Chatbot 的区别

| 维度 | Chatbot | Agent |
|---|---|---|
| 核心目标 | 对话回复 | 完成任务 |
| 流程 | 多数是输入到输出 | 计划、行动、观察、调整 |
| 工具使用 | 可以有，但通常是辅助 | 核心能力之一 |
| 状态管理 | 主要是对话历史 | 任务状态、工具结果、失败重试 |
| 适合场景 | FAQ、普通问答、客服闲聊 | 调研、自动化任务、复杂多步骤问题 |

一句话：

> Chatbot 主要负责“对话回复”，Agent 负责“围绕目标做决策和行动”。

## 什么样才算更自主的 Agent？

可以按自主程度分三档：

```text
1. Workflow
流程固定：A -> B -> C。
系统按预设节点走，LLM 主要负责某些节点里的生成。

2. Agentic Workflow
大流程固定，但有条件判断：
检索失败 -> fallback
JSON 失败 -> 重试
低置信 -> 人工审核

3. Autonomous Agent
给它一个目标，它能自己规划步骤、选择工具、观察结果、调整计划，直到完成或停止。
```

更完整的 Agent 通常要有这些能力：

```text
目标理解：知道用户最终要完成什么
规划能力：能把任务拆成多个步骤
工具选择：知道什么时候调用哪个工具
观察反馈：能读取工具结果
动态调整：结果不够时能换策略
状态管理：记住当前做到哪一步
停止条件：知道什么时候完成，什么时候放弃
安全护栏：高风险动作要人工确认
```

注意：

> Agent 不等于重新训练模型。Agent 是一种系统形态，可以用现成 LLM，加上工具、状态、工作流、权限和安全护栏。

## 结合 HSBC 项目怎么说

> 我的 HSBC 项目我不会夸成完全自主 Agent。更准确地说，它是 Agentic Workflow + RAG。因为整体流程还是 Dify Workflow 预设好的，比如预处理、规则检索、术语检索、fallback、JSON 校验；但其中有一些 agentic 特征，比如根据检索结果决定是否 fallback，根据 JSON 校验结果决定是否重试，根据低置信结果进入人工审核。所以它不是完全开放自主 Agent，而是带 Agentic 控制逻辑的 RAG 工作流。

## 这些算不算术语？

算，但更准确地说，它们是 Agent 的设计模式 / 运行机制术语，不是模型结构，也不是一定要重新训练模型。

先记住这四个层级：

| 概念 | 怎么理解 | 例子 |
|---|---|---|
| Agent | 一个围绕目标做决策和行动的系统 | 调研 Agent、客服 Agent、Coding Agent |
| 运行模式 / 设计模式 | Agent 做事的方法 | ReAct、Plan-and-Execute、Reflection |
| Tool | Agent 可以调用的单个能力 | 搜索、知识库检索、数据库查询、API |
| Workflow | 固定步骤的任务流程 | 预处理 -> 检索 -> 生成 -> 校验 |

一句话：

> Agent 是系统；ReAct、Plan-and-Execute、Reflection 是 Agent 的做事方法；Tool 是单个能力；Workflow 是固定流程。

## Agent 常见运行模式 / 设计模式

面试里不需要把每个模式讲成论文，先能区分适用场景就够了。

| 模式 | 一句话理解 | 适合场景 | 注意点 |
|---|---|---|---|
| ReAct | 想一步、做一步、看结果、再决定下一步 | 需要边查边做的开放任务 | 延迟高，容易循环 |
| Plan-and-Execute | 先制定整体计划，再按步骤执行 | 多步骤、目标比较明确的任务 | 计划可能过时，要能重规划 |
| Reflection / Reflexion | 执行后让模型评估哪里不好，再修正 | 写作、代码、复杂生成任务 | 反思不等于一定正确 |
| Workflow Agent / Agentic Workflow | 大流程固定，局部允许模型判断和重试 | 生产系统、低风险可控场景 | 自主性不如开放 Agent |
| Multi-Agent | 多个 Agent 分角色协作 | 研究、写作、审核、代码协作 | 成本高，协调复杂 |
| ReWOO | 先规划工具调用，再批量或并行执行 | 工具调用多、可并行的任务 | 灵活性比 ReAct 弱 |
| Tree of Thoughts | 同时探索多条思路，再选择更好的路径 | 推理、搜索、创意方案比较 | 成本高，不适合简单任务 |
| Human-in-the-Loop | 关键节点让人审核或确认 | 高风险、专业性强、低置信结果 | 会降低自动化程度 |

## 除了 ReAct，还有哪些常问？

### Plan-and-Execute

> Plan-and-Execute 是先让模型把任务拆成完整计划，再让执行器一步步完成。它和 ReAct 的区别是：ReAct 是边想边做，Plan-and-Execute 是先规划再执行。它适合目标明确、步骤较多的任务，比如做一份调研报告、执行一套数据处理流程。

风险是：

```text
计划错了，后面会跟着错。
```

所以工程上通常要加：

```text
- 每一步执行结果检查
- 失败后重新规划
- 最大步骤数
- 人工确认高风险步骤
```

### Reflection / Reflexion

> Reflection 是让 Agent 执行完之后，自己或借助另一个模型评估结果哪里不好，然后带着反馈再修正。它适合结果质量很难一次生成到位的任务，比如代码修复、长文生成、复杂问答。

注意：

```text
- Reflection 不是外部事实来源
- 模型反思也可能错
- 关键业务规则仍然要靠知识库、规则库或人工确认
```

### Multi-Agent

> Multi-Agent 是把任务拆给多个角色，比如 Researcher 负责查资料，Writer 负责生成，Reviewer 负责审核。它适合复杂协作任务，但不适合为了显得高级而乱用，因为多个 Agent 之间会有通信成本、冲突和错误传递。

面试里可以这么说：

> 如果任务需要明显的角色分工，比如检索、生成、审核互相独立，我会考虑 Multi-Agent。但如果只是一个固定转换流程，用单 Agent 或 Workflow 更稳、更便宜。

### Workflow Agent / Agentic Workflow

这是你 HSBC 项目最适合的说法。

> Workflow Agent / Agentic Workflow 不是让模型完全自由行动，而是在固定流程里加入一些模型判断和条件分支。比如检索结果不够就 fallback，JSON 校验失败就重试，低置信结果就转人工。

它的好处是：

```text
- 比普通 Workflow 灵活
- 比完全自主 Agent 可控
- 更适合生产里的专业场景
```

HSBC 项目可以接：

> 我们这个项目更接近 Agentic Workflow，而不是完全自主 Agent。因为主流程是 Dify Workflow 配好的，但里面有根据中间结果做 fallback、重试、校验和人工回流的控制逻辑。

### ReWOO

> ReWOO 可以理解成先把需要调用的工具和依赖关系规划出来，再批量执行。它不像 ReAct 那样每一步都等模型重新思考，所以在工具调用多、部分步骤可以并行时会更快。

面试里知道即可，不必主动展开，除非被追问 Agent 规划优化。

### Tree of Thoughts

> Tree of Thoughts 是让模型同时探索多条思路，再对不同路径打分或筛选。它适合需要试错和比较的推理任务，但成本较高，生产里不一定常用。

## 面试里怎么总括这些模式

可以背这一段：

> Agent 不是只有一种运行方式。ReAct 是边推理边调用工具，适合开放任务；Plan-and-Execute 是先规划再执行，适合多步骤任务；Reflection 是执行后反思修正，适合需要质量迭代的任务；Multi-Agent 是多个角色协作，适合复杂分工任务。生产里我更倾向于可控的 Agentic Workflow，把固定流程、工具调用、校验、重试和人工审核结合起来。

## ReAct 是什么？

### 15 秒版

> ReAct 是 Reasoning + Acting，也就是让 Agent 一边推理一边行动。它通常按 Thought、Action、Observation 循环：先思考下一步，调用工具，再观察工具结果，然后继续决定下一步。

### 60 秒版

> ReAct 是 Agent 里很常见的一种模式。它把模型的推理和工具调用结合起来，不是让模型一次性给最终答案，而是让它先判断需要什么信息，然后调用搜索、数据库、知识库或 API，拿到结果后再继续推理。
>
> 典型格式是：

```text
Thought：我需要先查某个信息
Action：调用 search / retrieval / database
Observation：工具返回结果
Thought：根据结果判断下一步
Action：继续调用工具或给出答案
```

> 它常用于 Agent，是因为很多任务不是靠模型内部知识就能完成的，需要外部工具和中间反馈。ReAct 能让模型把“想”和“做”连起来，降低瞎编概率，也让执行过程更可追踪。

## 为什么 Agent 里常用 ReAct？

核心原因：

```text
1. 支持工具调用
模型可以在需要信息时主动查工具，而不是纯靠记忆回答。

2. 支持多步任务
复杂任务可以拆成多轮：想一步、做一步、看结果、再决定下一步。

3. 降低幻觉
答案可以基于工具返回或检索结果，而不是完全凭模型猜。

4. 可观察
每一步工具调用和观察结果都能记录，方便 debug 和评估。

5. 能动态调整
第一次工具结果不够时，可以换 query、换工具或重试。
```

## ReAct 的局限

面试里不要只夸 ReAct，也要知道它的问题：

```text
1. 延迟高
每一轮 Thought/Action/Observation 都可能多一次 LLM 或工具调用。

2. 容易循环
如果没有停止条件，可能一直查、一直重试。

3. 错误会传导
前一步推理错了，后面的工具调用也会错。

4. 上下文变长
多轮 Observation 都塞进上下文，可能让模型混乱。
```

常见控制方法：

```text
最大步数 max_steps
工具超时 timeout
失败重试上限 retry limit
低置信转人工
结构化 tool schema
记录每一步日志
```

## 结合 HSBC 项目怎么说 ReAct

> HSBC 项目不是一个标准的完全开放 ReAct Agent，但可以借用 ReAct 的思想来理解：系统不是一次性生成 Gloss，而是会根据中间结果继续调整。比如第一次检索不够，就做 query rewrite 或 fallback retrieval；JSON 校验失败，就带着错误信息重试；术语不确定，就标低置信或进入专家审核。这些都体现了“观察中间结果，再决定下一步”的 agentic 思路。

更谨慎的说法：

> 我不会说项目完整实现了标准 ReAct Prompt，但它有类似 ReAct 的闭环思想：检索、观察、修正、再生成。

## 面试可背答案

> ReAct 是 Reasoning + Acting，核心是让 Agent 交替进行推理和行动。模型先 Thought 判断下一步需要什么，再 Action 调用工具，比如搜索、知识库或 API，然后根据 Observation 工具返回结果继续推理，直到完成任务。
>
> Agent 里常用 ReAct，是因为很多任务不能只靠模型内部知识。ReAct 让模型可以边查边做，基于外部结果调整下一步，也方便记录每一步，排查工具调用或推理错误。
>
> 但 ReAct 也有局限，比如多轮调用延迟高、容易死循环、前一步错误会传导。所以工程上要加最大步数、超时、重试上限、低置信转人工和日志追踪。

## 容易踩坑

不要说：

```text
ReAct 就是让模型一步步思考。
```

更准确：

```text
CoT 偏推理，ReAct 是推理 + 工具行动 + 观察反馈。
```

不要说：

```text
用了 RAG 就是 ReAct。
```

更准确：

```text
RAG 是检索增强生成；ReAct 是 Agent 的推理和行动循环。RAG 可以作为 ReAct 里的一个 Action。
```

一句话总结：

> ReAct 就是让 Agent “想一步、做一步、看结果、再决定下一步”。
