# Prompt 设计与结构化输出

这个文件用于准备 Prompt Engineering 高频问题。重点不是背“提示词技巧”，而是能说清楚：Prompt 如何服务一个真实 AI 应用，怎么约束模型、接入检索结果、稳定输出结构化结果，并通过测试集和专家反馈迭代。

## Prompt 具体怎么设计？

### 15 秒版

> 我设计 Prompt 一般会先明确任务、输入、规则、输出格式和异常处理。LLM 可以辅助起草 Prompt，但最终的规则、示例、输出 Schema 和评估标准要由人来定义，再通过测试集和 bad case 反复迭代。

### 60 秒版

> 我不会把 Prompt 理解成一句自然语言指令，而是把它当成一个小的任务说明书。一般会包含几个部分：角色和任务边界、输入字段、可用上下文或检索结果、必须遵守的业务规则、输出 Schema、few-shot 示例、禁止事项和错误处理。
>
> LLM 可以帮我生成 Prompt 初稿，或者根据 bad case 总结候选规则，但不能让它自己决定事实和业务规则。真正稳定的做法是：先用 Prompt 明确规则和格式，再用固定测试集跑结果，收集专家反馈，把常见错误回流到规则库、术语库、few-shot 示例或输出校验里。

### 项目结合版：HSBC HKSL Gloss

> 在 HSBC HKSL Gloss 项目里，我会把 Prompt 拆成几块。第一是角色定义：模型的任务不是自由翻译，而是把普通话/粤语文本转换成可供下游动画系统使用的 HKSL Gloss JSON。第二是输入说明：包括原始文本、语言类型、检索到的金融术语、HKSL 规则和可能的 unmatched phrases。第三是规则约束：金融术语和固定 Gloss 必须优先使用术语库结果，不允许模型自己编正式 Gloss；如果没有命中，就标低置信或进入人工审核。第四是输出格式：必须输出固定 JSON 字段，比如 gloss、unmatched_phrases、confidence、notes。第五是 few-shot 示例，给模型看几组“原句 -> HKSL Gloss JSON”的正确例子，也可以给反例说明什么不能做。
>
> 迭代上，我会用固定测试集看 Prompt 改动前后效果。如果专家反馈是术语错，就补术语库或别名；如果是语序错，就补 HKSL 规则和 few-shot；如果是格式错，就加强 Schema、JSON 校验和重试。Prompt 不是一次写完，而是和检索、规则库、测试集、专家反馈一起迭代。

## Prompt 的常见结构

```text
1. 角色定义
你是谁，要完成什么任务。

2. 任务边界
能做什么，不能做什么。

3. 输入字段
用户原文、语言、检索结果、规则、术语表等。

4. 规则优先级
术语库 > 规则库 > 模型常识；不确定就标记，不要编。

5. 输出格式
固定 JSON Schema、字段类型、必填字段。

6. Few-shot 示例
正确示例、边界示例、必要时加反例。

7. 异常处理
未命中术语、低置信、格式错误、需要人工审核。

8. 评估迭代
测试集回归、专家反馈、bad case 回流。
```

## 面试可背模板

> 我设计 Prompt 会按“任务说明书”的方式来写，而不是只写一句指令。先定义角色和任务边界，再说明输入字段和检索上下文，然后写清楚规则优先级，比如术语库结果优先、不能自由编 Gloss。接着约束输出 Schema，要求只输出 JSON，再放 few-shot 示例让模型学习语序和格式。最后设置异常处理，比如检索不到就标 low confidence 或进入人工审核。
>
> LLM 可以辅助起草 Prompt 和总结 bad case，但最终规则要来自业务知识、专家反馈和测试集验证。每次 Prompt 改动后，我会用固定测试集做回归，看 JSON 合法率、术语命中率、专家可用率有没有提升。

## 容易踩坑

不要说：

```text
Prompt 是让大模型自己写的。
```

更稳的说法：

```text
LLM 可以辅助起草和改写 Prompt，但人要定义任务边界、业务规则、输出结构和评估标准。
```

不要只说：

```text
我会给规则和示例。
```

要补成：

```text
我会给规则、示例、输出 Schema、禁止事项、异常处理，并用测试集和专家反馈迭代。
```

## 一句话总结

> Prompt 设计的核心不是写得长，而是把任务边界、上下文、规则优先级、输出结构和失败处理讲清楚，再用测试集不断修。
