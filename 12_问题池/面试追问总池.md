# 面试追问总池

这个文件用来收纳聊天里出现过的追问。它不是标准答案文件，而是复习路线图：哪些问题已经沉淀，哪些问题还要继续开口练。

## 使用方式

```text
已沉淀：说明已经写入项目卡、深挖问答、概念补基础或模拟面试记录。
待继续练：说明有初步口径，但还需要继续模拟面试。
谨慎口径：说明面试时不能说太满，避免过度包装。
```

## 待继续练的问题

下面这些问题要继续用模拟面试的方式练。优先级从上到下。

### 1. 为什么 Dify 足够，什么时候需要 LangGraph

核心要回答：

```text
Dify 适合快速验证业务流程。
LangGraph 适合更复杂状态管理、多 Agent、可测试性和工程可控性。
不要说 Dify 或 LangGraph 谁绝对更好，要说看阶段和需求。
```

### 2. 如果上线部署，工程化怎么补

核心要回答：

```text
FastAPI 接口
日志和监控
测试集回归
错误类型统计
人工审核队列
缓存和限流
Docker / CI/CD
```

### 3. 用户反馈效果差，怎么定位问题

核心要回答：

```text
先看输入是否复杂或口语化。
再看检索有没有命中。
再看 Prompt 是否正确使用检索结果。
再看 JSON 校验是否通过。
最后看专家反馈属于术语、语序、格式还是语义问题。
```

### 4. Prompt 具体怎么设计

核心要回答：

```text
角色和任务边界
输入字段
检索上下文
输出 Schema
Few-shot 示例
禁止自由编 Gloss
错误处理要求
```

## 谨慎口径

下面这些点面试时不要说太满。

```text
不要说“知识库是 LLM 生成的事实来源”。
更稳：LLM 辅助整理，事实来源是专家规则、术语表和反馈。

不要说“查不到就让模型自己编 Gloss”。
更稳：模型可以给候选草稿，正式 Gloss 必须经过术语库或专家确认。

不要主动说“用了 Milvus / pgvector”。
更稳：项目阶段主要用 Dify 内置知识库，后续可迁移。

不要把 20%+ 说成绝对精确值。
更稳：来自同一批 case 优化前后的专家可用性评估，具体数值按项目记录口径说。

不要把自己说成项目总架构师。
更稳：你参与 Prompt、知识库整理、Workflow 配置、测试和专家反馈闭环。
```
